Single-Drone System

Autonomous Drone System

  • An Autonomous Drone System is an unmanned aerial vehicle (UAV) capable of performing tasks with minimal or no human intervention.
  • It integrates hardware, software, communication protocols, and control algorithms into a single cyber-physical system.
  • The system continuously senses its environment, processes information, makes decisions, and executes control commands.
  • Modern autonomous drones are widely used in surveillance, mapping, agriculture, logistics, inspection, search and rescue, and military operations.

Characteristics and Functionalities

  • Characteristics:
    • Autonomous Navigation and sensor based perception
    • Real-time decision making and control
    • Wireless communication and mission execution
  • Functionalities:
    • Environment sensing and localization
    • State estimation and flight control
    • Data logging and path planning

ROS2 (Robot Operating System 2)

  • It is an open-source robotics middleware.
  • ROS2 enables the communication between distributed software components.
  • It provides standardized tools for robotic application development.
  • Its major components are Nodes, Publishers, Subscribers, Parameters, etc.

DDS (Data Distribution Service)

  • DDS is the communication middleware used by ROS2.
  • It removes the need for a central master server.
  • It supports real-time distributed applications
  • Its main features are automatic discovery, configurable QoS, security and scalability.

PX4 Autopilot

  • PX4 is an open-source flight control software for drones and other unmanned vehicles.
  • It provides a complete autopilot solution, including flight control algorithms, sensor drivers, and communication protocols.
  • PX4 supports various vehicle types, including multirotors, fixed-wing aircraft, and VTOLs.
  • It is compatible with various hardware platforms and integrates with ROS2 for advanced functionalities.

ROS2 and PX4 Integration

  • Sensor data are collected and processed by ROS2 nodes.
  • PX4 publishes the topics for communication with ROS2.
  • DDS transports the messages between the nodes.
  • ROS2 receives and processes the data for further analysis and control.
  • New commands are generated and sent back to PX4.

State Estimation and Mission Execution

  • State estimation involves determining the position, velocity, and orientation of the drone.
  • Mission execution refers to the autonomous navigation and task completion capabilities of the drone.

Limitations of Single Drone Systems

  • Limited coverage area and payload capacity.
  • Single point of failure, leading to mission failure if the drone malfunctions.
  • Limited scalability for large-scale operations.

Simulation Setup & Flight Guide

Now that you have successfully completed the workspace setup and tool configurations (Ubuntu, ROS 2, Gazebo, PX4, and QGroundControl), follow these detailed instructions to run a single-agent drone simulation and conduct an autonomous test flight.

Step 1: Launch PX4 SITL and Gazebo Environment

First, compile the PX4 firmware configuration specifically targeting Software-in-the-Loop simulation environments and automatically launch the quadrotor model into a 3D Gazebo environment.

Open a new Linux terminal window in Ubuntu using the system shortcut Ctrl + Alt + T, then change the environment directory to your local repository directory:

cd ~/PX4-Autopilot

Initiate the automated build pipeline and environment compilation target by entering the following command:

make px4_sitl gz_x500

Command Parameter Breakdown

  • make: Invokes the build automation system to compile the source code components.
  • px4_sitl: Defines the target output as a software-in-the-loop simulation environment.
  • gz_x500: Directs the simulator tool to spawn the standard x500 multirotor model framework inside Gazebo.

Compilation Note

  • Initial workspace builds can require additional runtime to map dependencies. Do not close this terminal instance, as it hosts your running PX4 shell window.

Step 2: Initialize QGroundControl Linkage

With the physical drone simulation initialized and active in your background server, connect a reliable ground console monitor. Open a secondary terminal instance.

Navigate to your executable directory storage point and launch the QGroundControl application layer script:

./QGroundControl.AppImage

Once initialization scripts settle, QGroundControl should seamlessly parse incoming MAVLink telemetry packet strings over your local interface structure. A voice announcement confirmation will announce status readiness, showing your active model structure tracking on the UI map layer.

Step 3: Execute Flight Tests

Autonomous physical flight control tracking states can be evaluated via two discrete methodologies within this framework configuration:

Method A: Graphical Mission Interface (QGroundControl)

  • Locate the "Takeoff" option within the top-left toolbar panel space.
  • Complete the operation by sliding the confirmation panel widget at the screen baseline.
  • Observe the Gazebo frame view to monitor engine spin-up routines and stabilization tracking during hovering maneuvers.
  • Select the corresponding interface "Land" control sequence parameter when ready to close down.

Method B: Direct Shell Terminal Control Utilities

Return attention to your initial working terminal window hosting the active pxh> interactive firmware environment string interface. Command direct vehicle control transitions using input strings:

commander takeoff

The system will lift off and assume standard automated holding limits within the simulation window. Bring the craft down smoothly by typing:

commander land
