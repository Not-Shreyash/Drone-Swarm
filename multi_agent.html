Multi-Drone System

Overview

  • A Multi-Drone System consists of two or more drones operating simultaneously within a shared mission environment.
  • Each drone functions as an independent robotic system while communicating and coordinating with other drones.
  • Multi-drone systems improve scalability, mission efficiency, redundancy, and coverage compared to a single drone.

Advantages of Multi-Drone Systems

  • Increased coverage area and improved fault tolerance.
  • Scalable and faster mission execution.

Single Drone vs Multi-Drone Comparison

Feature Single Drone Multi-Drone
Number of UAVs 1 2 or More
Coordination Not Required Required
Communication Internal Communication Only Internal + Inter-Drone Communication
Mission Planning Local Planning Centralized or Distributed Planning
Scalability Low High
Fault Tolerance Low (Single Point of Failure) High (Mission Continues if One Drone Fails)
Coverage Area Limited Large
Mission Completion Time Longer Shorter Due to Parallel Execution
Communication Complexity Low High
Deployment Cost Lower Higher
Maintenance Simpler More Complex
Typical Applications Inspection, Photography, Mapping Swarm Robotics, Search & Rescue, Delivery Fleets, Agriculture

Problems Introduced by Multiple Drones

  • The topic names might collide so it's important to use unique identifiers for each drone.
  • Coordination and control become more complex with multiple drones.
  • Managing communication and data sharing between drones requires robust protocols.

Simulation Setup Guide (Gazebo Harmonic)

The following structural framework defines how multiple independent vehicle targets interact and coexist within the Gazebo Harmonic execution layer.

Step 1: Compile the Modern SITL Target

Before executing deployment operations, verify that your regional PX4-Autopilot environment contains compiled targets mapped to the new Gazebo engine targets.

cd ~/PX4-Autopilot
make px4_sitl

Step 2: Launch Drone 1 (Starts the Simulator)

The first drone target instantiated parameters act as the core master runtime host environment. This initializes the primary Gazebo environment space window and drops the base quadcopter asset directly onto the local origin point layout.

Open a new terminal session (Terminal 1) and run:

cd ~/PX4-Autopilot
PX4_SYS_AUTOSTART=4001 PX4_SIM_MODEL=gz_x500 ./build/px4_sitl_default/bin/px4 -i 1

Environment Flags Breakdown:

  • PX4_SYS_AUTOSTART=4001 : Automatically initializes the system using the x500 baseline frame architecture definition.
  • PX4_SIM_MODEL=gz_x500 : Forces the deployment engine loop to bind and structure configurations onto the specialized Gazebo x500 SDF file layout.
  • -i 1 : Flags the framework to label this context as Instance 1, assigning dedicated internal multi-agent port mappings to preserve telemetry streams cleanly.

Step 3: Launch Drone 2 (Spawns into the Same World)

To drop secondary assets into the environment space, specify to the local engine workspace to skip opening redundant graphical client layers, binding directly into the running instance workspace via positional offsets.

Open a separate terminal session (Terminal 2) and run:

cd ~/PX4-Autopilot
PX4_GZ_STANDALONE=1 PX4_SYS_AUTOSTART=4001 PX4_GZ_MODEL_POSE="0,2,0" PX4_SIM_MODEL=gz_x500 ./build/px4_sitl_default/bin/px4 -i 2

Environment Flags Breakdown:

  • PX4_GZ_STANDALONE=1 : Critical flag parameter configuring the runtime environment to reuse the active simulation engine portal, preventing workspace execution faults.
  • PX4_GZ_MODEL_POSE="0,2,0" : Declares precise coordinate translation parameters [X, Y, Z] to offset the secondary asset by 2 meters, ensuring clean asset loading without model overlap.
  • -i 2 : Explicitly sets parameters to denote Instance 2, systematically separating MAVLink ports out of the path of Instance 1.

Step 4: Run the Micro XRCE-DDS Agent

To enable clean communication with ROS 2 layouts, activate the DDS middleware bridge. The internal network hooks monitor and manage multiple active clients simultaneously under a unified portal layer.

Open an additional terminal session (Terminal 3) and run:

MicroXRCEAgent udp4 -p 8888

The processing logs will quickly report successful initialization parameters tracking operational streams for both client 1 and client 2 assets.

Step 5: Verification and Telemetry Auditing

QGroundControl Dashboard

Launch QGroundControl within your system workspace. The application interface handles tracking of both distinct MAVLink streams automatically. The upper dashboard interface screen highlights an options drop-down element listing Vehicle 1 and Vehicle 2, allowing distinct mission operational tracking profiles.

ROS 2 Interface Nodes

Confirm isolated namespace handling arrays by initializing a distinct shell workspace and evaluating running active cluster topics:

ros2 topic list

The architecture layer automatically formats all corresponding telemetry and diagnostic streams under isolated namespaced structures, preventing cross-talk tracking metrics:

  • /px4_1/fmu/out/vehicle_odometry
  • /px4_2/fmu/out/vehicle_odometry

Assigning Custom Unique Namespaces

By default, the PX4 simulator uses your -i (instance) flag to generate generic prefixes like px4_1 and px4_2.

If you want semantic, custom names (e.g., drone_alpha and drone_beta), you can inject the PX4_UXRCE_DDS_NS environment variable before launching your targets.

Close your current simulation terminals and restart them with these commands:

Terminal 1 (Drone Alpha - Master Launch)

cd ~/PX4-Autopilot
PX4_UXRCE_DDS_NS=drone_alpha PX4_SYS_AUTOSTART=4001 PX4_SIM_MODEL=gz_x500 ./build/px4_sitl_default/bin/px4 -i 1

Terminal 2 (Drone Beta - Spawn into Same World)

cd ~/PX4-Autopilot
PX4_UXRCE_DDS_NS=drone_beta PX4_GZ_STANDALONE=1 PX4_SYS_AUTOSTART=4001 PX4_GZ_MODEL_POSE="0,2,0" PX4_SIM_MODEL=gz_x500 ./build/px4_sitl_default/bin/px4 -i 2

Terminal 3 (Micro XRCE-DDS Agent)

MicroXRCEAgent udp4 -p 8888

Custom Namespace Verification

Open a separate terminal and audit your active ROS 2 topics. First, source ROS 2 in your current terminal workspace:

If you are running ROS 2 Jazzy:

source /opt/ros/jazzy/setup.bash

(If you aren't sure which distribution you installed, type ls /opt/ros/ to check the directory folder name).

Once you've run the correct source command, review your active ROS 2 topics:

ros2 topic list

You will see that the generic prefixes have been replaced with your targeted custom namespaced variants:

  • /drone_alpha/fmu/out/vehicle_odometry
  • /drone_beta/fmu/out/vehicle_odometry
